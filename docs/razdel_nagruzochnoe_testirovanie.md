# 3.4 Нагрузочное тестирование веб-приложения агентства недвижимости

Нагрузочное тестирование (performance testing) — вид нефункционального тестирования, направленный на оценку поведения программной системы при увеличении интенсивности обращений со стороны пользователей. В рамках дипломного проекта нагрузочное тестирование выполнялось для подтверждения того, что разработанное веб-приложение агентства недвижимости сохраняет работоспособность и отдаёт корректные HTTP-ответы при имитации одновременной работы нескольких посетителей на публичных страницах каталога. Тестирование не заменяет функциональные проверки бизнес-логики (модерация, договоры, роли), но дополняет их количественной оценкой времени отклика и устойчивости сервера.

Целью нагрузочного тестирования являлась проверка:

- доступности основных публичных маршрутов приложения под параллельными GET-запросами;
- времени отклика (латентности) при различной степени параллелизма;
- отсутствия сбоев (HTTP 5xx, обрыв соединения) на сценариях, имитирующих просмотр каталога объявлений, главной страницы и вспомогательных разделов.

Объектом тестирования выступало веб-приложение на платформе Laravel 12 (PHP 8.2+), реализующее каталог объявлений, личный кабинет, модерацию, договоры и административную панель. В нагрузочных сценариях проверялись только **публичные** маршруты, не требующие аутентификации, поскольку они формируют основной поток обращений неавторизованных пользователей и наиболее показательны для оценки «витрины» системы.

---

## 3.4.1 Постановка задачи и критерии оценки

Для учебного стенда были приняты следующие критерии успешности прогона:

1. **Доля успешных ответов** — не менее 95 % запросов с кодом HTTP 2xx (на практике ожидался результат 100 % при отсутствии сетевых сбоев).
2. **Отсутствие ошибок сервера** — коды HTTP 500, 502, 503 не допускались.
3. **Время отклика** — фиксировались минимальное, среднее, 50-й (медиана, p50), 95-й (p95) и 99-й (p99) перцентили в миллисекундах; сравнивались сценарии с низкой и повышенной параллельностью.
4. **Сравнение с одиночным пользователем** — дополнительно измерялась латентность при последовательных запросах без конкуренции за ресурс PHP-процесса.

Критерии согласованы с типовой практикой оценки веб-сервисов: для интерактивных страниц ориентиром часто служит медиана порядка 200–500 мс на промышленном окружении; для локального сервера разработки допустимы более высокие значения, что отражается в разделе об ограничениях методики.

---

## 3.4.2 Средства и программная реализация тестов

Для воспроизводимости результатов в каталоге `tools/` проекта размещены скрипты:

- `load_test.php` — основной нагрузочный прогон с параллельными HTTP-запросами через расширение PHP `curl` и API `curl_multi`;
- `load_test_sequential.php` — контрольный замер латентности при последовательных обращениях;
- `load_test_report.json` — машиночитаемый отчёт последнего прогона (дата, параметры, таблица метрик по каждому URL).

Параметры прогона задаются переменными окружения `LOAD_CONCURRENCY` (число одновременных соединений в пакете) и `LOAD_REQUESTS` (общее число запросов на один URL). Базовый адрес сервера передаётся аргументом командной строки, по умолчанию `http://127.0.0.1:8000`.

Запуск приложения выполнялся командой:

```
php artisan serve --host=127.0.0.1 --port=8000
```

Данная команда поднимает встроенный HTTP-сервер PHP для разработки. Он обрабатывает запросы **в одном рабочем процессе**, что принципиально отличается от промышленной схемы nginx + PHP-FPM с пулом воркеров и должно учитываться при интерпретации задержек под параллельной нагрузкой.

## 3.4.5 Методика проведения испытаний

Последовательность действий при проведении нагрузочного тестирования:

1. Установка зависимостей проекта (`composer install`, настройка `.env`, миграции БД).
2. Запуск веб-сервера разработки на порту 8000.
3. Выполнение сценария S1: `php tools/load_test.php http://127.0.0.1:8000` с параметрами по умолчанию (10 / 50).
4. Выполнение сценария S2 с переменными `LOAD_CONCURRENCY=25`, `LOAD_REQUESTS=100`.
5. Контрольный прогон `php tools/load_test_sequential.php` — по 10 последовательных запросов к `/`, `/properties`, `/login`.
6. Сохранение агрегированных результатов в `tools/load_test_report.json` и перенос в пояснительную записку.

Для каждого URL рассчитывались: число успешных ответов, распределение HTTP-кодов, min, avg, p50, p95, p99 (мс). Под «успешным» понимался ответ с кодом 200–399.

### Сравнительный анализ S1 и S2

Увеличение параллельности с 10 до 25 и числа запросов с 50 до 100 привело к росту медианы времени отклика примерно в **2,4–2,6 раза** (например, `/properties`: p50 1879,6 мс → 4875,7 мс). Такое поведение характерно для **однопоточного** встроенного сервера: параллельные клиенты фактически становятся в очередь, и измеренная латентность отражает не только время обработки SQL и шаблона, но и время ожидания освобождения единственного worker-процесса.

Сопоставление таблиц 3.4 и 3.6 показывает разрыв между «идеальным» одиночным запросом (~195 мс на каталог) и медианой при стрессе (~4876 мс): порядка **25-кратного** увеличения обусловлен архитектурой стенда, а не обязательно дефектом прикладного кода при текущем объёме данных (4 активных объявления).

## 3.4.7 Ограничения методики и достоверность результатов

При включении раздела в пояснительную записку необходимо явно указать ограничения, влияющие на обобщение выводов:

1. **Тип веб-сервера.** `php artisan serve` не предназначен для эксплуатации в production; результаты демонстрируют устойчивость логики приложения и БД на учебном стенде, но не верхнюю границу производительности промышленной конфигурации.

2. **Режим отладки.** `APP_DEBUG=true` увеличивает объём диагностики и может незначительно влиять на время ответа.

3. **Объём данных.** При четырёх объявлениях нагрузка на СУБД минимальна; при росте каталога до сотен и тысяч записей целесообразно повторить тесты с индексами по полям фильтрации (`status_obyavleniya_id`, `tip`, `operatsiya`, `tsena`) и пагинацией.

4. **Состав сценариев.** Не тестировались POST-операции (создание объявления, вход, модерация), генерация PDF-отчётов, обращения к API DaData — они создают иные профили нагрузки (CPU, внешние сети).

5. **Сеть.** Тесты выполнялись на localhost, без задержек WAN и балансировщиков.

Несмотря на ограничения, методика воспроизводима: скрипты и JSON-отчёт входят в состав проекта, что соответствует требованиям к проверяемости результатов в дипломной работе.

## 3.4.8 Выводы по результатам нагрузочного тестирования

По итогам сценариев S1 и S2 можно сформулировать следующие выводы:

1. **Функциональная устойчивость под нагрузкой.** При суммарно 1050 HTTP-запросах к публичным страницам (350 + 700) приложение не вернуло ошибок сервера; все ответы имели код 200. Критерий доступности выполнен полностью.

2. **Время отклика в условиях одного пользователя** находится в диапазоне 150–250 мс для проверенных маршрутов, что для локального стенда с MySQL и Blade-рендерингом является приемлемым.

3. **Поведение под параллельной нагрузкой** на встроенном сервере характеризуется ростом латентности до 1,5–5,3 с (p50–p95) без отказов. Наиболее чувствителен к нагрузке маршрут каталога `/properties`.

4. **Прикладной уровень** (Laravel, запросы к БД, шаблоны) при малом объёме данных не выявил деградации в виде ошибок; узкое место учебного прогона — **модель обработки запросов одним процессом PHP**, а не отказ MySQL.

5. **Рекомендации для промышленного развёртывания:** использование nginx (или аналога) и PHP-FPM с пулом воркеров; отключение режима отладки; кэширование конфигурации и представлений (`php artisan config:cache`, `view:cache`); при высокой посещаемости — перенос сессий в Redis; повторное нагрузочное тестирование после наполнения БД реалистичным объёмом объявлений.

Таким образом, разработанное веб-приложение агентства недвижимости **выдерживает имитированную нагрузку на публичные страницы без потери работоспособности** на тестовом стенде. Для подтверждения соответствия требованиям по времени отклика в production-среде целесообразно провести повторные испытания на Open Server Panel или аналогичном стеке (Apache/nginx + PHP-FPM + MySQL) с теми же скриптами и, при необходимости, специализированными средствами (Apache JMeter, k6).
